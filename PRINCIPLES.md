## Methodological Discipline (project-specific, validated)

### P1 — No Look-Ahead Bias

All backtests and detector evaluations must use information strictly
available at the bar of decision. Confirmation at `bar_index + W` is
the only sanctioned future-looking operation, and only for swing
detection (lag is explicit). Audit document : `docs/NO_LOOKAHEAD_AUDIT.md`.

### P2 — Friction-Realistic Backtesting

Trade-by-trade friction model is mandatory: Hyperliquid taker fee
4.5 bps × 2 legs + tiered slippage (BTC/ETH 3.5 bps median, mid-caps
11 bps, ADA/SOL 22 bps) + funding cost `1e-5 / hour`. No headline
performance number leaves the project without the **4-qualifier line**:
`METHODO · FRICTION · WINDOW · REGIME`. Implemented in
`strategies/icc_cycle.py:_apply_friction`.

### P3 — Walk-Forward OOS Gating

Any strategy parameter is validated only by **walk-forward
out-of-sample** windows (12m train / 6m test / 3m step). In-sample
numbers are reported only with explicit `[in-sample, do not trust]`
tagging. Implemented in `strategies/walkforward_icc.py` and
`scripts/walkforward_v1_friction.py`.

### P4 — Test Unitaires as Foundations

Existing unit tests are **load-bearing**. A test that passed before
must continue to pass. Tests may be **extended** (new cases, stricter
assertions) but never **weakened**. Removal requires explicit
`@deprecated` marker with date + reason. Cf. message Badoun 23 May 2026
*"on peut ajouter un avenant à des TU pour les renforcer, mais on ne
les modifie ou efface JAMAIS directement"*.

### P5 — No Mid-Trip Parameter Changes (Principle 18)

When Badoun is in absence mode (paper trading deployment), **no
parameter of the production detector or daemon is modified**. Diagnostic
documentation only. Decisions to calibrate deferred to the next
synchronous gate (3 June 2026 for the current window, then 12 August
2026 for Phase 2).

### P6 — Belief State Transparency

State of belief in the project is tracked explicitly with both
**construction stage** % (what's built) and **belief stage** % (how
confident we are it works). The two are never confused. Current
state per Saturday Recap latest snapshot.

## Universal Principles (Operator Methodology v1.0, cross-project)

Adopted 26 May 2026 across Trading Bot V2, Synapse, and other agents
under Sebastien Assohou's operatorship.

### P7 — Compression Discipline

In mature phases of a project, **prefer compression over expansion**.
Reports, documents, and outputs should reduce surface area, not
proliferate. New artifacts must justify why they cannot be a section
update of an existing artifact.

Tactical applications:
- Saturday Recaps are **single page** maximum
- Status reports prefer **one diagram + one paragraph** over multi-page
  exposition
- When tempted to write a new doc, first check if 2-3 paragraphs in an
  existing doc would suffice

### P8 — Pattern Naming

When a recurring operational or methodological behavior is observed
≥ 2 times, it must be **named** and documented. Named patterns become
shorthand that compresses team communication.

Examples already named in this project:
- "4-qualifier line" (P2)
- "Belief stage vs construction stage" (P6)
- "Principle 18" / "No mid-trip parameter change" (P5)
- "Test Unitaires as fundations" (P4)
- "Range zone retroactive validation" (`STRATEGIC_LOGIC_DOC.md` concept 12)

### P9 — Layered Inquiry

When a complex question arrives (architectural decision, ambiguous
spec, strategic choice), **ask between layers**. Don't jump to the
answer. Layer 1: clarify the question. Layer 2: surface assumptions.
Layer 3: identify constraints and conflicts. Layer 4: propose 2-3
options with explicit trade-offs. Layer 5: recommend.

This is the protocol behind the "🟡 to validate" markers in
`STRATEGIC_LOGIC_DOC.md` and the conflict matrix.

## Operational Discipline (process)

### P10 — Saturday Recap

Every Saturday at 12:00 UTC, a Recap is generated covering the past
week. Format prescribed in `OPERATOR_METHODOLOGY.md` Section III.
Generated by `scripts/generate_saturday_recap.py`, saved in
`WEEKLY_RECAPS/YYYY-MM-DD_recap.md`, TL;DR sent via Telegram.

### P11 — Source Citation

Every numeric claim in any artifact (deck, dossier, recap, message)
carries an explicit source. Sources include: commit hashes, exchange
APIs with timestamps, dated documents. No hand-waving "as we saw" or
"earlier estimates" — point to the artifact.

### P12 — Hibernation Protocol

When Badoun is offline (absence period), V2 operates in **hibernation
mode**: monitoring only, no mutations to strategy, daemon runs
unattended with watchdog. Specified in `OPERATOR_METHODOLOGY.md`
Section IV.

### P13 — Automatic Integration of Strategic Updates (META)

When new information of strategic significance arrives — external
feedback at high quality, mentor guidance, paradigm-shifting insight,
or an idea that meaningfully improves the project's architecture or
operating model — the agent **integrates it automatically**.
Documentation is updated; new patterns are named; new principles are
adopted if warranted; sibling projects are notified; the change is
timestamped and attributed. **No explicit instruction is required.
The reflex is structural.**

Operationalization on V2 :
- Update `OPERATOR_METHODOLOGY.md` if the update extends the universal
  methodology
- Update `STRATEGIC_LOGIC_DOC.md` if the update touches a detection
  or filtering concept
- Update `IDEAS_PHASE_2.md` if the update is a new strategy candidate
  (parked for Phase 2 gate)
- Update `PRINCIPLES.md` if the update establishes a new operating
  discipline
- Snapshot prior versions in the corresponding `_history/` folder
- Log a session note in `docs/RECAPS/` documenting origin + date +
  what was integrated
- Notify the operator (Telegram) of what was integrated, not asking
  permission

Reference : `OPERATOR_METHODOLOGY.md` Pattern 13.

### P14 — Look-Ahead Bias Exclusion in SL Placement

At the moment a trade is placed, **only information available at that
exact bar is used to calibrate the SL**. Highs, lows, swing extensions,
or volatility readings that form **after** the entry bar are never
incorporated into the SL — even retrospectively in backtests, paper
runs, or post-mortems.

This is the trading-execution specialization of P1 (No Look-Ahead Bias).
Cross-reference: **Synapse Principle 18** (the parent no-look-ahead
discipline shared across all projects).

Operationalization on V2 :
- SL reference price is computed from data at `bar_index ≤ entry_bar`.
- The "swing cluster top" (Concept 14 of `STRATEGIC_LOGIC_DOC.md`) is
  identified using only confirmed swing points (W-lagged) available at
  entry time.
- In backtests : the SL must be set at the same bar as the entry
  decision, never "tuned" using subsequent bars (would be the textbook
  retrospective look-ahead bias).
- In paper trading and live : SL is computed at order submission, not
  recomputed using bars that arrived since.

Audit reference : `docs/NO_LOOKAHEAD_AUDIT.md` (to be extended with a
SL-specific section after the 3 June calibration meeting).

**Phase 1 SL buffer table (finalized by Sebastien 29 May 2026, fixed
until gate 12 August 2026)** :

|---|---:|---|
| Gold (MGCM) | **+$5** | = 5 points × $1 ≈ $50 P&L on micro contract (~0.111% spot) |

- OB- → `SL = swing_cluster_top + buffer[asset]`

où il sera le plus à l'aise. Optimisation Phase 2 si nécessaire."*

### P15 — Backup-Before-Action / Reversibility Discipline

Toute action qui modifie un état doit être précédée d'un checkpoint
explicite permettant le retour. **Pas de rollback = pas d'action.**

Cross-reference : Synapse Principle 31 — codifiée par Sebastien le
21 juin 2026, adoptée transversalement sur tous les projets.

Application V2 :
- Git commit propre AVANT toute modification de stratégie, config
  trading, ou paramètre PnL-sensitive
- `bash scripts/v2_snapshot.sh <label>` AVANT toute modif `live/state/*`,
  `strategies/*.py`, `paper_trading/*.py`. Capture : git HEAD + status
  + diff + stash + branch + daemon_state + heartbeat + trades.jsonl
  + processes ps + ROLLBACK.md procédure 5-step embedded
- Snapshots persistés dans `snapshots/SNAPSHOT_<UTC_TS>_<label>/`,
  jamais purgés sans approval explicite
- Paper mode validation 24h MINIMUM avant tout déploiement live ou
  changement de paramètres PnL-impact
- JAMAIS d'exécution de trade order live sans confirmation explicite
  opérateur (déjà règle existante via guardrail financial-actions,
  renforcée par P15)
- Pour migration VPS Hetzner : dry-run 24h en parallèle Mac AVANT
  cutover live, jamais de big bang. Cutover atomique <5 min hors
  funding hours UTC, snapshot pre/post obligatoire, rollback procedure
  documentée et testée AVANT cutover

### P16 — Weekly Reports Append-Only

Chaque dimanche (ou prochaine interaction après dimanche), V2 écrit un
weekly report dans `~/Desktop/trading-bot-v2/weekly_reports/` avec
naming `weekly_report_YYYY-MM-DD.md` (date = dimanche de fin de semaine
UTC). **Append-only** : jamais écrasé, chaque dimanche = nouveau fichier.

Format standardisé en 7 sections :
1. Summary (2-3 phrases) — état live + sprint progress
2. Key metrics — equity (realized + funding accrued), open positions,
   cycle count + uptime, funding events bookés
3. Actions completed — commits + sprints, bullets
4. Issues / Blockers
5. Plans next week
6. P15/P31 compliance — `v2_snapshot.sh` runs (count + labels),
   branches actives
7. Memorable phrase / quote

Codifié par Sebastien (via Dispatch) le 23 juin 2026 — protocole
transversal aux agents. Premier report V2 : `weekly_report_2026-06-22.md`
(semaine 16-22 juin, audit lookahead-fix + migration VPS Phase 1).

Operationalization : V2 génère le report autonomement le dimanche soir
ou à la première interaction post-dimanche. Le fichier est commité
dans le repo via le batch commit hebdomadaire. Les fichiers passés sont
préservés intacts (append-only).
