## 1. OB- (Order Block bearish)

**Définition humaine (Badoun)** :
> *"OB- = la DERNIÈRE bougie haussière avant le mouvement baissier qui
> finalise le LL. Si pendant le move baissier une autre bougie haussière
> apparaît, elle devient la NOUVELLE référence OB- (parce qu'elle est la
> plus récente avant la continuation baissière qui finalise le break)."*

**Définition mathématique** :
- Pré-condition : cassure structurelle baissière confirmée (NEW_LOW ou LL,
  cf. concepts 4 et 9) sur la fenêtre considérée
- Bougie candidate `i` : `close[i] > open[i]` strictement (skip dojis)
- Recherche : walk-backward depuis `break_bar - 1` jusqu'à `origin_bar_index`
  du break (le swing high précédant le swing low cassé — cf. concept 8)
- Sélection : **la bougie `i` la plus récente** dans cette plage qui
  satisfait `close[i] > open[i]`
- Ancres :
  - `wick_anchor_price = low[i]` (référence d'invalidation immédiate)
  - `body_close = close[i]` (référence du niveau "validé")
  - `zone = [low[i], high[i]]` (zone wick-inclusive)

**Conditions d'invalidation (l'OB- n'est PAS déclenché)** :
- Aucune cassure NEW_LOW/LL dans la fenêtre → pas de trigger structurel
- Aucune bougie strictement haussière entre le break et son origin → walk-back vide
- L'OB- se trouve dans une **range zone non cassée** (cf. concept 11) → invalidé
- Bias macro contraire (cf. concept 17) → noté mais non tradé (cf. concept 18)

**Priorité** : 1. En cas de conflit avec concept 11 (range zone), le filtre
range **prime** : un OB- in_range n'est pas exploitable même si techniquement
détecté, sauf si concept 12 (breakout retro) s'applique.

**Timeframe(s) concerné(s)** : H4 principalement (Badoun's TF de référence
pour Gold/ETH/BTC). À tester aussi sur H1 et M30 en Phase 2.

**Sources** :
- Message Badoun Cowork 22 mai 2026 (correction définition dynamique) : *"OB- = la DERNIÈRE bougie haussière avant le mouvement baissier qui finalise le LL"*
- Screenshot `badoun_screenshots/00_btc_annotated.jpeg` (validation OB- bar 28 cerclé)
- Screenshot `badoun_screenshots/01_eth_annotated.jpeg` (3 OB- top validés implicitement)
- Code de référence : `strategies/ob_detector_v2_dynamic.py:detect_obs_dynamic`

**Statut** : ✅ Validé par Badoun.

## 3. CHoCH (Change of Character)

**Définition humaine (Badoun)** :
> Pas de définition verbatim explicite reçue. Inférence basée sur le contexte
> ICC standard : un CHoCH est le **premier signe de retournement de tendance**
> — la première cassure d'une structure qui contredit la séquence dominante
> (ex : dans un downtrend de LH/LL, le premier HH est un CHoCH haussier).

**Définition mathématique (proposée)** :
- Tendance dominante actuelle = séquence des N dernières structures
  (`LH→LL→LH→LL` = downtrend ; `HH→HL→HH→HL` = uptrend)
- CHoCH haussier : première occurrence d'un `HH` (close > précédent swing
  high) dans une séquence dominée par LH/LL
- CHoCH baissier : première occurrence d'un `LL` dans une séquence dominée
  par HH/HL
- Confirmation : Badoun applique la "double cassure" (cf. concept 5) — un
  CHoCH unique n'est pas suffisant pour confirmer un retournement

**Conditions d'invalidation** :
- Cassure de très petite amplitude (fake breakout) — typiquement gérée
  par le concept 5
- CHoCH dans une range zone (concept 11) — invalidé
- CHoCH sans displacement institutionnel (concept 10) — non valide

**Priorité** : 1.

**Timeframe(s) concerné(s)** : H4 pour la stratégie, mais le CHoCH doit
être interprété dans le contexte du TF supérieur (D1 si possible) — cf.
bias macro concept 17.

**Sources** :
- Inférence ICC standard (Inner Circle Concepts par "The Inner Circle
  Trader") — terminologie héritée du SMC
- Concept 5 (double cassure) implique que Badoun connaît et utilise CHoCH
- Pas de citation directe encore

**Statut** : 🟡 Proposé par V2. **Question ouverte critique** : comment
Badoun distingue-t-il opérationnellement un CHoCH d'un BoS ? La distinction
ICT/SMC classique est que CHoCH est un retournement et BoS est une continuation.
À clarifier au 3 juin.

## 4. BoS (Break of Structure)

**Définition humaine (Badoun)** :
> Pas de définition verbatim explicite reçue. Inférence ICC standard :
> un BoS est la **continuation d'une tendance existante** — la cassure
> d'un swing point dans le sens dominant (ex : dans un uptrend, un nouveau
> HH qui dépasse le précédent HH est un BoS).

**Définition mathématique (proposée)** :
- Tendance dominante = séquence HH/HL (uptrend) ou LH/LL (downtrend)
- BoS haussier : nouveau HH dans un uptrend déjà établi (≥ 2 HH/HL précédents)
- BoS baissier : nouveau LL dans un downtrend déjà établi (≥ 2 LH/LL précédents)
- Distinction CHoCH (concept 3) : CHoCH = 1er retournement ; BoS = continuation

**Conditions d'invalidation** :
- Cassure dans une range zone (concept 11)
- Cassure sans displacement (concept 10)
- Cassure trop proche du swing précédent (fake breakout — concept 5)

**Priorité** : 1.

**Timeframe(s) concerné(s)** : H4 principalement.

**Sources** :
- Inférence ICC standard
- Mention "CHoCH/BoS" dans le message Badoun du 23 mai : *"Le mouvement
  après la bougie OB doit créer un nouveau swing low (pour OB-) ou un
  nouveau swing high (pour OB+) ... structural break (CHoCH/BoS) en aval"*

**Statut** : 🟡 Proposé par V2. **Question ouverte** : pour V2-dyn, les

## 5. Double cassure ICC

**Définition humaine (Badoun)** :
> Nouvelle règle Badoun aujourd'hui : *"pour éviter fake breakouts, ICC
> demande 2 cassures pas 1."* (citation transmise par Cowork 23 mai)

**Définition mathématique** :
- Un break structurel (NEW_LOW/LL ou NEW_HIGH/HH) n'est confirmé qu'après
  **2 closes** au-delà du swing point cassé
- Variante stricte : 2 H4 closes consécutifs au-delà du niveau
- Variante tolérante : 2 H4 closes dans une fenêtre de 5 H4 (le 2e peut
  être après une mèche de retracement)

**Conditions d'invalidation (le break n'est PAS confirmé)** :
- Un seul close au-delà puis retracement → **fake breakout**, OB associé
  invalidé
- Le 2e close revient en-deçà du swing point (`close[i+1] <= swing_price`)

**Priorité** : 1. **Probablement le filtre le plus important pour éliminer
les faux positifs en zone range** (concept 11). Surface beaucoup de
détections actuelles de V2-dyn comme fake breakouts.

**Timeframe(s) concerné(s)** : H4. Très probablement applicable aussi sur
H1/M30 si on étend à ces TFs en Phase 2.

**Sources** :
- Message Badoun 23 mai 2026 (transmis via Cowork)
- Cohérent avec littérature ICT : *"double break confirms"* est une heuristique
  reconnue

**Statut** : ✅ Validé par Badoun. **Question d'implémentation ouverte** :
2 closes consécutifs OU 2 closes dans une fenêtre de N bars ? À préciser.

## 6. HH (Higher High)

**Définition humaine (Badoun)** :
> Pas de définition verbatim mais implicite : un HH est un swing high qui
> dépasse le swing high précédent **par close** (pas par mèche).

**Définition mathématique** :
- Détection swing : `is_swing_high(prices, i, w)` — `high[i] > high[i-k]` ET
  `high[i] > high[i+k]` pour `k in [1, w]`
- Promotion HH : si le close `close[i]` ou le high `high[i]` du nouveau
  swing high dépasse le précédent swing high broken
- Variante Badoun probable : utiliser le `close` plutôt que le `high` pour
  éliminer les mèches manipulatrices (à confirmer 3 juin)
- Cf. concept 10 (displacement) : un HH sans body large = potentiel fake

**Conditions d'invalidation** :
- Bougie isolée (high > voisins mais close revient sous niveau précédent)
  → potentielle manipulation, pas HH valide
- HH dans une range zone (concept 11)
- HH sans validation 2-closes (concept 5)

**Priorité** : 1.

**Timeframe(s) concerné(s)** : H4 principal. Cohérence multi-TF (HH H4
peut être contredit par LH D1 — cf. concept 17 bias macro).

**Sources** :

## 10. Displacement institutionnel

**Définition humaine (Badoun)** :
> *"il n'y a pas de cassure évidente, pas de bougies avec un vrai mouvement
> institutionnel — évidemment certaines bougies seront plus grandes que
> d'autres mais ça ne veut pas dire qu'il y a eu injection de capitaux
> sur les marchés"*.

Le **displacement** est la signature visuelle d'une intervention de
capital significative : une ou plusieurs bougies avec body large
relativement au range moyen, dans une direction nette, accompagnées
d'un volume notable (si disponible).

**Définition mathématique (proposée)** :
- Calcul à un instant `t` :
  - `atr_50 = ATR(50)` (volatility baseline)
  - `body_size = |close[t] - open[t]|`
  - `total_range = high[t] - low[t]`
- Indicateurs de displacement (au moins **2 sur 3** doivent être vrais) :
  - `body_size > 1.5 × atr_50` (body large vs baseline)
  - `body_size / total_range > 0.6` (body domine la mèche — momentum directionnel)
  - `body_size > avg_body_5 × 1.5` (body significativement > moyenne récente)

**Conditions d'invalidation (pas de displacement)** :
- Bougie principalement constituée de mèches (`body_size / total_range < 0.4`)
- Body inférieur à l'ATR récent → bruit
- Bougie isolée suivie d'un retracement immédiat (consommation > 70% sur
  la bougie suivante)

**Priorité** : 1. Le displacement est un **pré-requis pour valider une
cassure structurelle comme exploitable** (cf. concepts 3, 4, 11).

**Timeframe(s) concerné(s)** : doit être évalué sur le TF de l'OB (H4
pour la stratégie principale). Cohérence multi-TF souhaitable : un
displacement H4 confirmé par displacement D1 vaut beaucoup plus.

**Sources** :
- Citation Badoun 23 mai (transmise via Cowork) : *"injection de capitaux
  sur les marchés"*
- Concept fondamental SMC/Wyckoff : *"smart money leaves footprints"*
- Cohérence avec concept 11 : un range zone est l'absence de displacement
  pendant N bars

**Statut** : 🟡 Proposé par V2. **Question ouverte** : les 3 indicateurs
proposés sont une heuristique de V2 — Badoun valide-t-il les seuils
(1.5×ATR, 0.6 body/range ratio) ? Sinon, quels seuils utiliser ?

## 11. Range zone (Concept 1 Badoun)

**Définition humaine (Badoun)** :
> *"tant que ce n'est pas cassé par une autre structure, les OBs qui se
> créent dedans sans casser le prix dans ce carré ne sont pas valides.
> Si dans ce carré un OB se forme et casse soit en haut soit en bas →
> cet OB sera valide. Tant que ça reste dedans → c'est du range."*

Une range zone est une zone de **consolidation entre 2 swing points
structurels**, dans laquelle le prix oscille sans direction franche.
Tant que le prix reste dans la box, tous les OBs internes sont **non
exploitables**, même si techniquement détectés. Un breakout du range
(cf. concept 12) revalide rétroactivement les OBs internes.

**Définition mathématique** :
- Identification d'une range zone :
  - `range_top` = swing high récent non broken
  - `range_bottom` = swing low récent non broken
  - Le prix oscille entre `range_top` et `range_bottom` pendant ≥ N bars
    (typiquement N ≥ 5 sur H4)
  - Absence de displacement institutionnel (cf. concept 10) sur les N bars
  - Body candles moyens : `avg(body_size) / atr < threshold` (≤ 0.7)
- Test "OB in range" pour un OB à bar `i` :
  - `range_bottom ≤ low[i] ≤ range_top` ET `range_bottom ≤ high[i] ≤ range_top`
- État du range : `active` (non cassé), `broken_up`, `broken_down`

**Conditions d'invalidation (la zone n'est PAS un range)** :
- Au moins 1 displacement institutionnel détecté dans les N bars (concept 10)
- Variance des closes > seuil (mouvement directionnel)
- Body candles moyens > seuil ATR
- Cassure d'un des bords par un close confirmé (cf. concept 5 : 2 closes)

**Priorité** : 1. **Filtre N°1 anti-faux-positifs**. Dans l'exercice
alignment du 22-23 mai : 7/13 OBs V2-dyn rejetés par Badoun étaient dans
des range zones (ETH 4/7, Gold 3/5).

**Timeframe(s) concerné(s)** : H4 pour la stratégie. À considérer aussi
en multi-TF — un range H4 peut être un mini-swing D1.

**Sources** :
- Citation directe Badoun (annotation manuscrite + Cowork 23 mai)
- Screenshots :
  - `badoun_screenshots/01_eth_annotated.jpeg` (1 grand rectangle orange 18-22 mai)
  - `badoun_screenshots/02_gold_annotated_v1.jpeg` (rectangle orange middle)
  - `badoun_screenshots/03_gold_structure_ranges.jpeg` (3 rectangles orange superposés)

**Statut** : ✅ Validé par Badoun (3 screenshots avec annotations explicites).
**Implémentation** : non codée, restera à coder pour V3. Cf. `proposed_v3_output_format.md`.

## 12. Breakout retroactive validation (Concept 2 Badoun)

**Définition humaine (Badoun)** :
> Si un OB se forme dans une zone de range MAIS que le prix finit par
> casser au-dessus ou en-dessous du range → l'**OB devient valide
> rétrospectivement**. La zone reste "range" tant qu'elle est respectée,
> devient "trigger zone" quand elle est cassée.

**Définition mathématique** :
- Pour chaque OB initialement classé `in_range = True` (concept 11) :
  - Tracker à chaque nouvelle bougie post-OB l'état du range englobant
  - Si `close[t] > range_top` (avec confirmation 2 closes, cf. concept 5) →
    `range_broken_direction = up`, re-évaluer
  - Si `close[t] < range_bottom` → `range_broken_direction = down`, re-évaluer
  - Lors de la re-évaluation :
    - Si `OB- in_range` ET `range_broken_direction = down` → **OB- devient valide
      retro** (le range a finalement cédé dans le sens du OB-)
    - Si `OB+ in_range` ET `range_broken_direction = up` → **OB+ devient valide
      retro**
    - Sinon → OB reste invalidé (range broken dans la mauvaise direction)
- Flag explicite dans l'output V3 : `retro_validated = True`

**Conditions d'invalidation (l'OB ne devient PAS retro-valide)** :
- Range cassé dans la direction opposée à l'OB
- Range cassé sans displacement (faux breakout)
- Cassure non confirmée par 2 closes (concept 5)
- Délai trop long entre formation OB et cassure range (heuristique : > 20
  bars sur H4 ?) — à valider 3 juin

**Priorité** : 2 (important pour le recall, mais pas critique pour le MVP).

**Timeframe(s) concerné(s)** : H4. Re-évaluation continue à chaque nouvelle
bougie.

**Sources** :
- Citation Badoun 23 mai (transmise via Cowork)
- Cas concret : ETH OB- bar 48 (`badoun_screenshots/01_eth_annotated.jpeg`)
  est dans le range orange initialement, mais l'OB- manuel que Badoun
  ajoute à droite hors range probablement résulte d'un breakout retroactive
- Diagnostic V2 dans `eth/v2_reasoning_per_ob.md` (section OB MANQUÉ +A)

**Statut** : ✅ Validé par Badoun.
**Questions ouvertes** :
- Quel délai max entre OB et cassure pour rester valide ? (timeout retro)
- Si plusieurs OBs in_range, tous sont-ils retro-validés en chaîne, ou
  seulement le plus récent ?

## 13. Jambe structurelle (Concept 3 Badoun)

**Définition humaine (Badoun)** :
> *"tant que ce n'est pas cassé par une autre structure, les OBs qui se
> créent dedans sans casser le prix dans ce carré ne sont pas valides."*

Concept lié au range mais distinct : une **jambe structurelle** est un
segment de prix entre 2 swing points consécutifs qui montre une direction
franche (séquence HH/HL = jambe haussière, LH/LL = jambe baissière).
**Seuls les OBs formés dans une jambe valide sont exploitables**, pas
ceux entre 2 swings successifs qui forment un range.

**Définition mathématique** :
- Classification de chaque OB à bar `i` :
  - Identifier la séquence des N derniers swing points (8-10 dernières structures)
  - Si la séquence montre directionalité claire :
    - HH puis HL puis HH puis HL → jambe haussière `leg_type = "HH-HL"`
    - LH puis LL puis LH puis LL → jambe baissière `leg_type = "LH-LL"`
  - Si la séquence alterne (HH puis LH, LL puis HL, etc.) → `leg_type = "transition"` ou `leg_type = "in_range"`
- `leg_origin_bar` = début de la jambe en cours
- `leg_break_bar` = bar où la jambe s'est cassée (= dernière cassure
  structurelle)
- Test exploitabilité OB : `leg_type in ("HH-HL", "LH-LL")` (pas transition,
  pas in_range)

**Conditions d'invalidation (l'OB n'est PAS dans une jambe valide)** :
- `leg_type = "in_range"` (cf. concept 11)
- `leg_type = "transition"` (les 2-3 structures précédentes alternent
  sans direction)
- Jambe trop courte (< 2 swing points dans la direction) → pas encore
  établie

**Priorité** : 1. **Filtre N°2 anti-faux-positifs** (complémentaire au
filtre range). Souvent recouvrant : un OB in_range est typiquement aussi
dans une jambe transition.

**Timeframe(s) concerné(s)** : H4. Cohérence multi-TF souhaitable.

**Sources** :
- Citation Badoun (idem concept 11)
- Screenshot `03_gold_structure_ranges.jpeg` : Badoun a tracé en blanc/gris
  la structure complète Gold (high mai 12 → low → high mai 14 → low mai 18
  → high mai 19 → low) et marqué les ranges en orange
- Diagnostic dans `divergence_report.md` section "Concept 3"

**Statut** : ✅ Validé par Badoun (screenshot explicite avec traits).
**Question ouverte** : combien de swing points minimum pour qualifier
une jambe ? 2 (HH + HL) ou 3 (HH + HL + HH) ?

## 14. SL placement vs OB placement (Concept 4 Badoun)

**Définition humaine (Badoun, annotation manuscrite)** :
> *"Bougie qui est techniquement le LH mais l'OB valide est juste en
> dessous, je te l'ai mis. Du coup s'il y a une entrée en Sell le SL
> devrait se trouver au dessus de la bougie de la structure, celle avec
> la flèche orange."*

La **bougie qui marque le LH structurel** (swing point au top du leg)
et la **bougie qui est le vrai OB exploitable** sont souvent **deux
bougies différentes**. Le SL se place au wick top de la **bougie
structurelle**, pas de la bougie OB.

**Définition mathématique** :
- Pour chaque OB détecté :
  - `ob_candle_bar` = la bougie OB selon concept 1 ou 2 (most recent
    opposite avant break)
  - `structural_swing_bar` = le swing point structurel qui a engendré le
    LH (pour OB-) ou HL (pour OB+) — peut être différent de `ob_candle_bar`
  - **Sortie V3 split** :
    - `ob_entry_zone.high` = `max(open, close)` de `ob_candle_bar`
    - `ob_entry_zone.low` = `min(open, close)` de `ob_candle_bar`
    - `ob_entry_zone.anchor` = `low` de `ob_candle_bar` (OB-) ou `high` (OB+)
    - **`sl_reference_price`** = `high` de `structural_swing_bar` (OB-)
      ou `low` (OB+) — séparé !
- Recherche `structural_swing_bar` : walk-back depuis `ob_candle_bar`
  jusqu'à trouver le swing high (OB-) ou swing low (OB+) le plus proche
  qui définit le LH structurel du leg

**Conditions d'invalidation (la séparation ne s'applique pas)** :
- `ob_candle_bar` EST déjà le swing structurel (ob candle = structural
  swing) → `sl_reference_price = ob_entry_zone.anchor`, pas de séparation
- Le swing structurel est trop éloigné (> 5 bars sur H4) → reconsider
  si on est vraiment dans le même leg

**Priorité** : 1. **Améliore drastiquement le risk:reward réel**. Dans
l'exemple Gold (screenshot 04) :
- OB entry à 4555, SL au wick OB seul (4588) = R seul de 33 pts
- OB entry à 4555, SL au wick LH structurel = identique ici (4588) car
  ils coïncident
- Cas plus typique (OB candle ≠ structural swing) : R proper évite les
  stop-outs sur micro-mèches

**Timeframe(s) concerné(s)** : H4 (le TF du leg). SL peut être ajusté
en fonction du TF supérieur si volatilité D1 le justifie.

**Sources** :
- Annotation manuscrite Badoun screenshot `04_gold_sl_vs_ob_placement.jpeg`
- Détail dans `proposed_v3_output_format.md` (schéma `EntryZone` +
  `sl_reference_price` séparés)

**Statut** : ✅ Validé par Badoun (annotation manuscrite explicite).
**Implémentation** : nouvelle fonction `find_structural_swing_for_sl`
à écrire pour V3.

## 15. FVG strict vs override "??" subjective

**Définition humaine (Badoun)** :
> Première version Badoun ETH alignment : *"je préfère louper une entrée
> que de rentrer sans savoir pourquoi"*. Décision : **pas de "??" dans
> V2/V3** — précision prime sur recall.

> Mais : la définition FVG elle-même reste **stricte** : un gap explicite
> entre bougie i et i+2 (cf. ICC textbook).

**Définition mathématique (FVG)** :
- Bearish FVG entre bougies `i` et `i+2` : `high[i+2] < low[i]` strictement
- Bullish FVG : `low[i+2] > high[i]` strictement
- **Pas de tolérance epsilon** initialement (sera reconsidéré 3 juin si
  trop de faux négatifs)

**Politique override "??"** :
- Badoun annote parfois "OB-??" sur ses screenshots quand son intuition
  visuelle signale un OB potentiel mais le critère technique ne passe
  pas — typiquement quand le push directionnel est fort mais le FVG est
  absent et consec < 5
- **V2/V3 ne code PAS ces "??"** : le bot ne traite que les OBs validés
  par critère strict
- Les "??" de Badoun sont notés comme info qualitative dans les datasets
  d'apprentissage (cf. `data/ob_labels/`) mais ne déclenchent aucune
  action algorithmique

**Conditions d'invalidation (FVG non détecté)** :
- `high[i+2] >= low[i]` (les wicks se touchent ou se chevauchent) → pas de FVG
- Décision : pas de gap → pas de validation FVG (mais cassure structurelle
  + displacement peuvent suffire — cf. concept 5)

**Priorité** : 1 pour la définition stricte FVG. 1 pour la règle "no ??".

**Timeframe(s) concerné(s)** : H4 principalement.

**Sources** :
- Citation Badoun 22 mai : *"je préfère louper une entrée que de rentrer
  sans savoir pourquoi"*
- Décision Cowork 22 mai sur exclusion des "??"
- Code de référence : `strategies/ob_detector_v2_dynamic.py:_detect_fvg_in_move`

**Statut** : ✅ Validé par Badoun. **Question ouverte** : la définition
"OB- = bougie pré-FVG" suggérée dans `gold/v2_reasoning_per_ob.md`
(Question 7) — la bougie OB doit-elle être celle juste avant le FVG, ou
peut-elle être plus loin si plusieurs bougies opposites avant le FVG ?

## 16. Zone de prix protégée par les acheteurs/vendeurs

**Définition humaine (Badoun)** :
> Inférence d'un message Badoun passé : *"si le prix descend en grignotage
> sans gros volume d'opposition et atteint 2000, puis rebond fort vers
> 2300 → 2000 est probablement une zone protégée par les acheteurs"*.

Une **zone protégée** est un niveau de prix où les acheteurs (zone
support) ou vendeurs (zone résistance) défendent activement, identifiable
par la **réaction asymétrique** du prix : descente lente vs rebond rapide
(pour un support), ou montée lente vs chute rapide (pour une résistance).

**Définition mathématique (proposée)** :
- Identifier des niveaux candidats :
  - Pour un support : `level` = prix de turnaround où le marché a renversé
    après baisse
- Caractériser la "descente lente" :
  - N bars baissières avec body moyen < 0.6 × ATR
  - Volume moyen baissier < volume moyen haussier précédent
  - Pas de displacement institutionnel baissier (concept 10) sur les N bars
- Caractériser le "rebond fort" :
  - Bougie de reversal avec body > 1.5 × ATR (= displacement haussier)
  - Confirmation par 2-3 bougies haussières suivantes
- Niveau protégé = `low[reversal_bar]` (pour support) ou `high[reversal_bar]`
  (pour résistance)

**Conditions d'invalidation (zone non protégée)** :
- Symétrie : descente rapide ET rebond rapide → pas de protection asymétrique
  identifiable
- Niveau ré-atteint puis cassé → la protection a cédé, zone obsolète
- Délai trop long entre protection et test → l'effet décay (heuristique :
  > 100 bars sur H4 ?)

**Priorité** : 2. **Complémentaire aux OBs**, pas critique pour le MVP.
Peut servir de confluence d'entrée ou de filtre macro pour les OBs.

**Timeframe(s) concerné(s)** : D1 et H4 principalement. Une zone protégée
D1 reste valide sur des semaines.

**Sources** :
- Message Badoun passé (date à reconstituer)
- Concept SMC classique : *"liquidity pools and protected highs/lows"*

**Statut** : 🟡 Proposé par V2 sur la base d'inférence. **À clarifier au
3 juin** :
- Citation verbatim Badoun à retrouver
- Définition exacte de "grignotage sans gros volume" (heuristique
  proposée — quel seuil ?)
- Lien opérationnel avec OBs : confluence d'entrée, filtre, ou stratégie
  séparée ?

## 17. Bias macro (alignment avec tendance dominante)

**Définition humaine (Badoun, inférée du screenshot 01 ETH)** :
> Implicite. Sur les screenshots ETH et BTC, Badoun X-out systématiquement
> les **OB+ contre-trend** dans un downtrend macro. Donc règle implicite :
> un OB+ dans un downtrend macro = noté mais pas tradé. Symétriquement OB-
> contre-trend dans uptrend macro.

**Définition mathématique (proposée)** :
- Bias macro à un instant `t` :
  - Calculé sur un TF supérieur (D1 si on trade H4) — ou sur la même TF
    avec lookback long
  - Indicateurs candidats :
    - `close[t] > MA200` → bias haussier
    - `close[t] > MA50 > MA200` → bias haussier confirmé
    - Symétrique pour bias baissier
    - Séquence des 3-5 dernières structures (HH/HL = bull, LH/LL = bear)
    - Pente de la régression linéaire des closes sur les 50 dernières bars
- Décision sur un OB :
  - OB- + bias bear → **exploitable** ✓
  - OB+ + bias bull → **exploitable** ✓
  - OB- + bias bull → noté, **pas tradé** (cf. concept 18)
  - OB+ + bias bear → noté, **pas tradé**

**Conditions d'invalidation (bias non déterminable)** :
- Bias mixte (MA50 et MA200 contradictoires)
- Régression linéaire avec R² < 0.3 → tendance trop bruitée
- Période de transition (CHoCH récent non confirmé)

**Priorité** : 1. Devrait éliminer **tous les OB+ contre-trend détectés
par V2-dyn dans l'exercice alignment** (BTC bars 27 et 47, ETH bars 33
et 47). Compte : 4 OB+ rejetés / 6 OB+ détectés = 67% des OB+ étaient
contre-trend.

**Timeframe(s) concerné(s)** :
- Stratégie principale : H4 pour les OBs
- Bias macro : **D1** (cohérent avec la cascade ICC Daily/H4/H1)
- Le bias H4 peut servir de secondaire dans certains cas

**Sources** :
- Inférence visuelle des screenshots `00_btc_annotated.jpeg` et
  `01_eth_annotated.jpeg`
- Pattern systémique de rejet OB+ dans les diagnostiques `btc/v2_reasoning_per_ob.md`
  et `eth/v2_reasoning_per_ob.md`
- Concept SMC standard : "trade with the higher TF bias"

**Statut** : 🟡 Proposé par V2. **Question ouverte au 3 juin** :
- Quel indicateur exact Badoun utilise pour déterminer le bias ?
- Sur quel TF ?
- Combien de "force" minimum pour un bias clair ?

## 18. Validation OB vs tradabilité OB (distinction critique)

**Définition humaine (Badoun)** :
> Distinction Badoun implicite via le pattern d'annotation : **un OB
> peut être structurellement valide mais non tradable**. Citation
> indirecte : Badoun annote certains OBs avec "OB-" sans X (= valides
> structurellement) mais ne les liste pas comme cibles tradables (parce
> que contre-trend, dans range, etc.).

**Définition opérationnelle** :
- **OB valide** : passe les critères techniques (concepts 1-5, 8-9, 15)
- **OB tradable** : passe en plus les filtres contextuels (concepts 11,
  13, 17)

Un OB peut donc avoir 3 statuts :
1. **Invalide** : ne passe même pas les critères techniques (rejeté avant
   classification)
2. **Valide mais non tradable** : passe les critères techniques mais pas
   les filtres contextuels (in_range, in_transition, bias contraire)
3. **Tradable** : passe tous les critères + filtres → signal d'entrée
   réel

**Définition mathématique** :
```
ob.is_valid_technically = (
    structural_break_confirmed  # concept 5
    AND ob_candle_found         # concepts 1/2
    AND fvg_or_displacement     # concepts 10/15
)

ob.is_tradable = (
    ob.is_valid_technically
    AND not ob.is_in_range_unbroken    # concept 11
    AND ob.in_valid_leg                # concept 13
    AND ob.bias_macro_aligned          # concept 17
)
```

**Conditions d'invalidation (devient non tradable)** :
- Un seul filtre contextuel échoue → `is_tradable = False`
- Re-validation possible si le contexte évolue (concept 12 retro)

**Priorité** : 1. **Conceptuellement fondamental** : sépare les OBs notés
(pour mémoire, dataset, alignment exercise) des OBs actionnables (signaux
d'entrée).

**Timeframe(s) concerné(s)** : H4 pour les OBs. Bias macro D1.

**Sources** :
- Pattern d'annotation Badoun (les "non commentés" et les ✅ sans X dans
  les screenshots)
- Discussion implicite dans le diagnostic BTC : pourquoi V2-dyn detect 6
  et Badoun valide partiellement — distinction valide vs tradable
- Citation paraphrasée : la décision sur les "??" implique cette distinction

**Statut** : ✅ Validé par Badoun (implicite mais cohérent dans tous les
diagnostics).
**Implémentation** : 2 flags séparés dans la sortie V3 (`is_valid_technically`
et `is_tradable`).

## 19. Multi-layer OB (Macro / Refined / Entry)

**Définition humaine (Badoun)** :
> Validé par Badoun aujourd'hui (23 mai, transmis via Cowork) — pas de
> citation verbatim disponible, mais le concept SMC standard a 3 niveaux
> hiérarchiques.

**Définition opérationnelle (proposée)** :
- **OB Macro** (TF D1) : niveau structurel large, sert de zone de
  référence pour le bias macro et le SL global
- **OB Refined** (TF H4) : niveau intermédiaire, principal terrain
  d'opération du bot V2/V3
- **OB Entry** (TF H1 ou M30) : raffinement pour le timing d'entrée
  précis dans la zone OB Refined

**Définition mathématique** :
- Hierarchie : OB Entry ⊂ OB Refined ⊂ OB Macro
- Cohérence : un OB Entry n'est valide que si :
  - Il est inclus dans la zone d'un OB Refined (intersection des wick zones)
  - L'OB Refined est lui-même dans la direction d'un OB Macro proche ou
    aligné avec le bias macro
- Détection : runner le détecteur OB sur 3 TFs (D1, H4, H1) et établir
  les correspondances

**Conditions d'invalidation (multi-layer rompu)** :
- OB Entry hors zone OB Refined → suspect, à ignorer
- OB Refined sans OB Macro support → tradable mais avec confidence
  réduite (concept 17 bias)
- 2 layers en désaccord directionnel (OB+ refined dans OB- macro) →
  contre-trend, non tradable

**Priorité** : 2. **Raffinement avancé** — utile mais le MVP V3 peut
fonctionner avec H4 seul (concept 19 = nice-to-have post-MVP).

**Timeframe(s) concerné(s)** :
- OB Macro : D1
- OB Refined : H4
- OB Entry : H1 ou M30

**Sources** :
- Confirmation Badoun 23 mai (transmise via Cowork)
- Concept SMC standard (ICT teaches "Daily → 4H → 1H → 15M cascade")
- Cohérent avec les "TF à tester" de IDEAS_PHASE_2.md concept #2
  (H4/H1/M30, H8/H4/H1)

**Statut** : 🟡 Proposé par V2. Confirmation Badoun reçue mais pas de
détails opérationnels. **Questions ouvertes au 3 juin** :
- Quelle hiérarchie exacte de TFs Badoun préfère ? (D1/H4/H1 ou H4/H1/M30 ?)
- Comment formaliser la "cohérence" entre layers ? Intersection stricte
  ou tolérance ?
